**Test 2: `e2e_cli_render_auth_failure_exits_nonzero`**

Verifies roadmap success criterion: "`sss render FILE` exits non-zero with a stderr message when keystore auth fails"

Strategy: Create a project with User A's key. Generate a different (User B) keypair in a separate isolated HOME. Add ONLY User B to the project. Attempt `render` with User A's keys → auth failure because User A is not in the project's authorized users. This exercises the `load_project_config_with_repository_key` auth failure path in `src/config.rs`.

Use the existing `generate_other_pubkey()` helper function (already defined in the test file) to get a second user's public key.

```rust
#[test]
fn e2e_cli_render_auth_failure_exits_nonzero() {
    // Set up a project, seal a file, then reinitialize the project
    // with a DIFFERENT user's key so the current user is no longer authorized.
    let env = SssTestEnv::new();
    env.setup(); // generates keys for "testuser", inits project with testuser

    // Seal a file using the authorized testuser
    env.write_file("secret.txt", "pass: \u{2295}{topsecret}");
    env.run_ok(&["seal", "--in-place", "secret.txt"]);

    // Now get a second (other) user's public key
    let other_pubkey = generate_other_pubkey();

    // Reinitialize the project with only the other user authorized
    // (sss init with a different username replaces the project config)
    let _ = env.cmd()
        .args(["init", "otheruser", &other_pubkey])
        .output()
        .expect("run init for other user");

    // Now "testuser" (our current keypair) is no longer in the project.
    // render should fail with an auth error.
    let (stdout, stderr) = env.run_fail(&["render", "secret.txt"]);
    assert_eq!(
        stdout, "",
        "render must produce no stdout on auth failure; got: {:?}",
        stdout
    );
    assert!(
        !stderr.is_empty(),
        "render must write an error message to stderr on auth failure"
    );
}
```

Note: If the `sss init <username> <pubkey>` syntax is different from what the above uses, read `src/commands/project.rs` or `src/main.rs` to find the correct argument structure for adding/replacing a project user, and adjust accordingly. The goal is: current keypair is NOT authorized for the project → render exits 1 with non-empty stderr.

**Test 3: `e2e_cli_non_interactive_flag_render`**

Verifies that the `--non-interactive` CLI flag (not just the `SSS_NONINTERACTIVE` env var) is accepted and produces clean stdout.

This matters because the research confirmed the flag works by setting `SSS_NONINTERACTIVE=1` internally. The Emacs integration guide should document the flag, so it must be tested as a flag.

The `SssTestEnv::cmd()` already sets `SSS_NONINTERACTIVE=1` via env. To test the CLI flag, use `bare_cmd()` which does NOT set that env var, then add `--non-interactive` manually:

```rust
#[test]
fn e2e_cli_non_interactive_flag_render() {
    let env = SssTestEnv::new();
    env.setup();
    env.write_file("data.txt", "api_key: \u{2295}{abc123}");
    env.run_ok(&["seal", "--in-place", "data.txt"]);

    // Use bare_cmd to avoid SSS_NONINTERACTIVE env var, add flag explicitly
    let out = env
        .bare_cmd()
        .args(["--non-interactive", "--kdf-level", "interactive", "render", "data.txt"])
        .output()
        .expect("run sss --non-interactive render");
    let stdout = String::from_utf8_lossy(&out.stdout);
    let stderr = String::from_utf8_lossy(&out.stderr);
    assert!(
        out.status.success(),
        "--non-interactive render must exit 0; stderr: {}",
        stderr
    );
    assert_eq!(
        stdout.trim(),
        "api_key: abc123",
        "--non-interactive render must output decrypted plaintext; got: {:?}",
        stdout
    );
    assert_eq!(
        stderr.trim(),
        "",
        "--non-interactive render must produce no stderr on success; got: {:?}",
        stderr
    );
}
```
