<objective>
Extend StoredKeyPair with optional hybrid fields and add the Keystore methods needed for dual-suite storage and retrieval.

Purpose: This is the data layer for Phase 3. Without it, the CLI handlers in Plan 03-02 have nothing to call. Every other plan in this phase depends on the contracts established here.

Output:
- `StoredKeyPair` gains two optional fields: `hybrid_public_key` and `hybrid_encrypted_secret_key`, both gated with `#[serde(default, skip_serializing_if = "Option::is_none")]` and `#[cfg_attr(not(feature = "hybrid"), serde(skip))]`.
- `Keystore::store_dual_keypair` handles all three suite paths: classic-only, hybrid-only-added-to-existing-classic, and fresh-both-atomically.
- `Keystore::load_hybrid_keypair` decrypts and returns a `HybridKeyPair` from a stored identity file.
- `Keystore::get_current_stored_raw` returns the raw `StoredKeyPair` for the current identity (needed by keys show in Plan 03-02 without full decryption).
- Integration tests covering the KEYSTORE-01 backward-compat, KEYSTORE-03 preserve-classic, and KEYSTORE-04 single-passphrase scenarios.
</objective>

Add three new public methods to `impl Keystore`, all gated with `#[cfg(feature = "hybrid")]`:

**`get_current_stored_raw`** — reads the raw StoredKeyPair for the current key without any decryption. Needed by `handle_keys_show` in Plan 03-02. Loads the current key ID, reads the TOML file, returns the StoredKeyPair.

**`store_dual_keypair`** — signature:
```rust
pub fn store_dual_keypair(
    &self,
    classic_keypair: Option<&ClassicKeyPair>,
    hybrid_keypair: Option<&HybridKeyPair>,
    password: Option<&str>,
) -> Result<String>
```

This method handles the three cases that `handle_keys_generate_command` will invoke:

Case A — `classic_keypair = Some`, `hybrid_keypair = Some`, no existing identity: generate fresh UUID, derive key once, encrypt both secrets with the same derived key (same salt). Write new TOML. Call `set_current_key`. Return key_id.

Case B — `classic_keypair = None`, `hybrid_keypair = Some`: caller is adding hybrid material to an existing identity. Read the current identity file (call `get_current_stored_raw`), verify `hybrid_public_key.is_none()` (else error "hybrid keypair already present in this identity"), append hybrid fields, write back to the SAME file path (do not generate new UUID, do not call `set_current_key`). This preserves the classic fields byte-for-byte (KEYSTORE-03).

Case C — `classic_keypair = Some`, `hybrid_keypair = None`: store classic only; this is identical to calling `store_keypair` for a `KeyPair::Classic`. Delegate to `store_keypair` by constructing `KeyPair::Classic(classic_keypair.clone())`.

For Cases A and B with a password, encode hybrid secret material using:
```rust
let hybrid_sk_b64 = base64::prelude::BASE64_STANDARD.encode(hybrid_kp.secret_bytes.as_ref());
```
This avoids copying secret bytes into a non-Zeroizing buffer (RESEARCH.md threat pattern). Then call `crate::crypto::encrypt_to_base64(&hybrid_sk_b64, &derived_key.to_encryption_key())`.

For the passwordless path, store the raw base64 directly (same warning as existing `store_keypair` passwordless path).

IMPORTANT for Case B: re-derive the KDF key using the EXISTING salt from the stored file (not a new salt). The stored file's `encrypted_secret_key` was encrypted under the original salt; you only need to derive the key to encrypt the new hybrid material under the same salt. Prompt the user for the passphrase in the handler (not here) — just accept `password: Option<&str>`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| disk → StoredKeyPair | Untrusted TOML from ~/.config/sss/keys/ is deserialized; malformed or tampered files cross here |
| passphrase → derived key | User-supplied string drives Argon2id; reuse of an existing salt for hybrid upgrade is the new path |
| HybridKeyPair::secret_bytes → TOML string | 1819 raw secret bytes must not touch a non-Zeroizing buffer before encryption |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Denial of Service | StoredKeyPair deserialization | mitigate | `#[serde(default)]` on both new Option fields; absence of hybrid fields in old files maps to None, not parse error. Mirrors the `in_keyring` precedent exactly. |
| T-03-02 | Information Disclosure | HybridKeyPair secret_bytes before encryption | mitigate | Encode directly from `secret_bytes.as_ref()` (avoids Vec copy); `Zeroizing<[u8;1819]>` drops secret on scope exit. Do not use `String::from(...)` or `.to_vec()` on secret material. |
| T-03-03 | Tampering | store_dual_keypair Case B overwrites existing file | mitigate | Read-verify-write pattern: assert `hybrid_public_key.is_none()` before write; apply `0o600` permissions after write (matches existing store_keypair pattern). |
| T-03-04 | Elevation of Privilege | `--suite hybrid` or `--suite both` without hybrid feature compiled | accept (Plan 03-02 concern) | Runtime gate in handler (`#[cfg(not(feature = "hybrid"))]` returns Err); schema layer does not need to enforce this — it only stores whatever the handler sends. |
| T-03-05 | Information Disclosure | load_hybrid_keypair reconstructed secret in plain array | mitigate | Use `Zeroizing::new(secret_array)` before returning; the `HybridKeyPair` struct field is already `Zeroizing<[u8; HYBRID_SECRET_KEY_SIZE]>` so reconstructed value auto-zeroizes on drop. |
</threat_model>
