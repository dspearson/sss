**CRITICAL accuracy rules from research:**
- Symmetric encryption: XChaCha20-Poly1305 via libsodium (NOT age crate)
- KDF: Argon2id via libsodium crypto_pwhash with ALG_ARGON2ID13 (NOT scrypt)
- Nonce derivation: BLAKE2b keyed hash (deterministic — same inputs produce same ciphertext)
- Key exchange: X25519 via crypto_box_seal (ephemeral sender key)
- Identity: Ed25519 keypairs
- Memory safety: zeroize crate for sensitive data

**Sections to include:**

1. **Overview** — One paragraph: sss uses marker-based encryption to protect individual secrets within files, not whole-file encryption. The surrounding context remains readable.

2. **Cryptographic Algorithms**
   - Table: algorithm, purpose, key size, source
   - XChaCha20-Poly1305: symmetric encryption of secret content (256-bit key, 192-bit nonce, 128-bit MAC)
   - BLAKE2b: deterministic nonce derivation (inputs: project_timestamp + file_path + plaintext; key: repository key)
   - X25519: asymmetric key wrapping via crypto_box_seal for multi-user key sealing
   - Ed25519: user identity keypairs
   - Argon2id: key derivation from passphrase (three levels)

3. **Key Derivation (Argon2id)**
   - Three levels with exact parameters from src/kdf.rs:
     - sensitive: OPSLIMIT_SENSITIVE (~4 iterations), MEMLIMIT_SENSITIVE (256 MiB) — default for keystore
     - moderate: OPSLIMIT_MODERATE (~3 iterations), MEMLIMIT_MODERATE (128 MiB)
     - interactive: OPSLIMIT_INTERACTIVE (~2 iterations), MEMLIMIT_INTERACTIVE (64 MiB)
   - Configurable via `--kdf-level` flag or `SSS_KDF_LEVEL` env var

4. **What IS Encrypted**
   - Content inside `⊠{...}` markers (the ciphertext payload)
   - Repository symmetric key in .sss.toml (sealed_key per user, via X25519 crypto_box_seal)
   - Private keys on disk (via Argon2id-derived key wrapping, when passphrase-protected)

5. **What is NOT Encrypted**
   - File structure and non-secret text surrounding markers
   - The presence of secrets (⊠{...} markers are visible to observers)
   - Filenames
   - Git history prior to sealing
   - Public keys in .sss.toml
   - Marker positions and count (an observer can see how many secrets a file contains)
   - .secrets file contents (these are plaintext, intended for local use only)

6. **Threat Model**
   - Protects against: secret values exposed when committed to git, files shared with untrusted parties, repository made public
   - Does not protect against: metadata analysis (number/position of secrets), timing attacks on large files, compromised system with memory access
   - DoS protection: 100MB per-secret limit (MAX_MARKER_CONTENT_SIZE), passphrase attempt rate limiting

7. **Deterministic Nonces**
   - Explain why: clean git diffs (same secret + same file = same ciphertext)
   - How: BLAKE2b keyed hash with project_timestamp + file_path + plaintext as input, repository key as the BLAKE2b key
   - Trade-off: an observer can detect whether a secret changed between commits (but cannot learn the content)

8. **Memory Protection**
   - zeroize crate: sensitive data (keys, plaintext) zeroed from memory after use
   - No plaintext caching in process memory beyond immediate need
   - Rate limiting on passphrase attempts

9. **Brief Plaintext Window**
   - Acknowledge: when using sss-mode (Emacs) or `sss edit`, plaintext exists on disk momentarily during the save-then-seal operation
   - This is identical to the epa-file.el pattern in Emacs
   - Mitigation: auto-save and backup files are disabled for decrypted buffers in sss-mode

Create docs/marker-format.md as the marker format reference. British English throughout. All claims must be verified against src/constants.rs and src/processor/core.rs.

**CRITICAL accuracy rules from research:**
- ⊕ (U+2295 CIRCLED PLUS) = plaintext/open marker, UTF-8: \xe2\x8a\x95
- ⊠ (U+22A0 SQUARE ORIGINAL OF) = ciphertext/sealed marker, UTF-8: \xe2\x8a\xa0
- ASCII alternative: o+{...} for plaintext input ONLY (no ASCII alternative for ciphertext)
- Parser uses balanced brace depth-counting (find_balanced_markers)
- Ciphertext payload: base64(nonce[24] || MAC[16] || ciphertext[N])

**Sections to include:**

1. **Overview** — sss uses inline markers to identify secret content within files. Two marker types: plaintext (user-authored) and ciphertext (machine-generated after sealing).

2. **Marker Syntax**
   - Grammar (BNF-style):
     ```
     plaintext-marker  ::= ("⊕" | "o+") "{" content "}"
     ciphertext-marker ::= "⊠" "{" content "}"
     content           ::= any byte sequence with balanced braces (depth-counted)
     ```
   - Plaintext markers: `⊕{secret}` or `o+{secret}`
   - Ciphertext markers: `⊠{base64ciphertext}`

3. **Character Reference**
   - Table with: symbol, Unicode name, codepoint, UTF-8 bytes, purpose
   - ⊕: U+2295 CIRCLED PLUS, \xe2\x8a\x95, plaintext/open marker
   - ⊠: U+22A0 SQUARED TIMES, \xe2\x8a\xa0, ciphertext/sealed marker
   - o+: ASCII (2 bytes), input alias for ⊕ (not accepted for ⊠)

4. **Balanced Brace Parsing**
   - The parser uses depth-counting: opening `{` increments depth, closing `}` decrements
   - The marker content ends at the first `}` where depth returns to 0
   - This means nested braces are valid: `⊕{{"key":"value"}}` — the outer braces are the marker, the inner braces are content
   - Example: `⊕{{"db": {"host": "localhost", "port": 5432}}}` is a single marker containing nested JSON
   - Size limit: MAX_MARKER_CONTENT_SIZE = 100 MB per marker

5. **Transformation Behaviours**
   - `sss seal`: `⊕{secret}` → `⊠{base64ciphertext}` (and `o+{secret}` → `⊠{base64ciphertext}`); existing `⊠{}` markers pass through unchanged
   - `sss open`: `⊠{base64ciphertext}` → `⊕{decrypted}` (always UTF-8 form, even if original was `o+`); existing `⊕{}` markers pass through unchanged
   - `sss render`: `⊠{base64ciphertext}` → `decrypted_plaintext` (no markers); `⊕{secret}` → `secret` (markers stripped)
   - Surrounding text is always preserved unchanged
   - Operations are idempotent on already-transformed markers

6. **Ciphertext Payload Format**
   - Inside `⊠{...}`, the content is base64-encoded binary:
     ```
     base64( nonce[24 bytes] || MAC[16 bytes] || ciphertext[N bytes] )
     ```
   - Nonce: 24 bytes, deterministically derived via BLAKE2b (not random)
   - MAC: 16-byte Poly1305 authentication tag
   - Ciphertext: XChaCha20 encrypted content (same length as plaintext)
   - Total binary overhead per secret: 40 bytes (nonce + MAC)
   - Total base64 overhead: ~54 bytes plus marker characters and braces

7. **Magic Detection**
   - A file whose content begins with the byte sequence `\xe2\x8a\xa0\x7b` (⊠{) is treated as containing sealed content
   - sss-mode uses this for automatic mode activation via magic-mode-alist

8. **Secrets Interpolation Marker**
   - `⊲{secret_name}` — references a value from a .secrets file
   - Resolved during seal/open operations before encryption
   - Not a marker in the same sense as ⊕/⊠ — it is replaced with the secret value, then the result can be wrapped in ⊕{} for encryption

9. **Implementing a Compatible Parser**
   - Pseudocode for the balanced brace algorithm
   - Edge cases: empty markers `⊕{}`, markers containing only whitespace, markers at EOF without trailing newline
