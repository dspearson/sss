Create docs/architecture.md as a technical architecture overview. British English throughout. This is NOT a replacement for the root-level ARCHITECTURE.md (which covers protocol spec) — this is a user-facing explanation of how sss works internally, aimed at contributors and advanced users.

**Read these source files before writing:**
- src/processor/core.rs — Processor struct, find_balanced_markers, process flow
- src/crypto.rs — encryption/decryption operations, key types
- src/config.rs — project key loading, user resolution

- src/fuse_fs.rs — FUSE filesystem overview

- src/secrets.rs — secrets file parsing

**Sections to include:**

1. **Overview** — sss is structured as a pipeline: input file → marker detection → transformation (seal/open/render) → output. The core is the Processor, which holds the repository key and performs marker-by-marker operations.

2. **Processor Pipeline**
   - Processor struct: holds RepositoryKey, project_root, SecretsCache
   - Three modes: seal (plaintext→ciphertext), open (ciphertext→plaintext), render (ciphertext→stripped plaintext)
   - Per-file operation: read file → find_balanced_markers → transform each marker → reassemble output
   - Diagram (ASCII art): File → Scanner → find_balanced_markers → [seal|open|render] → Output

3. **Marker Detection (find_balanced_markers)**
   - Two-pass approach: scan for marker prefixes (⊕, o+, ⊠), then depth-count braces to find closing brace
   - Brace depth counting algorithm: +1 for `{`, -1 for `}`, marker ends when depth returns to 0
   - Cross-reference to docs/marker-format.md for full syntax specification
   - Size limit: 100MB per marker (MAX_MARKER_CONTENT_SIZE)

4. **Key Loading Flow**
   - Step 1: Read .sss.toml from project root
   - Step 2: Find current user's section (from --user flag, SSS_USER env, or default_username in settings.toml)
   - Step 3: Load user's private key from keystore (~/.config/sss/keys/)
   - Step 4: Decrypt sealed_key from .sss.toml using user's private key (X25519 crypto_box_seal_open)
   - Step 5: RepositoryKey is now available for symmetric operations
   - Diagram (ASCII art): Keystore → Private Key → Unseal → Repository Key → Processor

5. **Marker Inference (8-step algorithm)**
   - Purpose: when editing a rendered file (markers stripped), sss must infer which parts were originally marked
   - Read the marker_inference/ directory to describe the steps at a high level
   - This is an advanced feature — link to source for full details

6. **Secrets Interpolation**
   - `⊲{secret_name}` syntax references .secrets files
   - Resolution: find .secrets file in project → look up key → substitute value
   - Multi-line values supported via pipe syntax
   - Resolved before seal/open, so the interpolated value becomes the marker content

7. **Project Scanner**
   - Walks project directory tree
   - Respects ignore patterns (.sss.toml ignore field, gitignore-style)
   - Used by --project flag operations

8. **FUSE Filesystem (Optional)**
   - Feature-gated: requires `--features fuse` at build time
   - Mounts a project directory with transparent rendering
   - Files appear decrypted at the mountpoint; originals remain sealed
   - Read-only and read-write modes
   - `sss git` command for passthrough git operations on underlying sealed files

9. **9P Server (Optional)**
   - Feature-gated: requires `--features ninep` at build time
   - Network-transparent file access via 9P protocol (Plan 9)
   - Three file views: `.file` (rendered), `.file.open` (with markers), `.file.sealed` (raw)
   - TCP and Unix socket support

10. **Module Map**
    - Brief description of each src/ module (draw from the existing README Code Structure section, but verify)
    - Group into: CLI layer (main.rs, commands/), core (processor/, crypto.rs, kdf.rs), storage (keystore.rs, config_manager.rs, project.rs), optional (fuse/, ninep_fs.rs), utilities

Create docs/sss-mode-guide.md as the sss-mode installation and usage guide. British English throughout. All details must be verified against emacs/sss-mode.el (354 lines).

**Read emacs/sss-mode.el completely before writing.** This is the single source of truth for all sss-mode behaviour.

**CRITICAL accuracy rules from research:**
- sss-mode passes `--non-interactive` to all CLI calls (confirmed from source)
- Passphrase must be in system keyring or SSS_PASSPHRASE env var (no TTY prompts in Emacs)
- Daemon-mode PATH issue: sss binary may not be on inherited PATH; use `sss-executable` customisation
- `sss keygen` is the actual subcommand called by sss-keygen command (deprecated but functional)
- `sss seal --project` is what sss-process calls (not `sss process`)
- Key bindings use C-c C-x pattern for package-lint compliance

**Sections to include:**

1. **Overview** — sss-mode is an Emacs major mode that provides transparent decrypt-on-open and re-seal-on-save for sss-sealed files. It is a single .el file with zero external dependencies.

2. **Prerequisites**
   - Emacs 27.1 or later
   - `sss` binary installed and accessible
   - Working keystore: `sss keys generate` must have been run
   - Project initialised: `sss init <username>` in the project directory
   - Passphrase accessible without TTY: system keyring OR `SSS_PASSPHRASE` env var

3. **Installation**
   - Clone repo or copy emacs/sss-mode.el
   - Add to init.el:
     ```elisp
     (add-to-list 'load-path "/path/to/sss/emacs/")
     (require 'sss-mode)
     ```
   - That's it — sss-mode registers itself on magic-mode-alist automatically

4. **Daemon-Mode / GUI Emacs Configuration**
   - Problem: when Emacs is launched from a GUI launcher or as a daemon, it may not inherit the shell PATH
   - Solution: set the absolute path explicitly:
     ```elisp
     (setq sss-executable "/usr/local/bin/sss")
     ```
   - Alternative: use exec-path-from-shell package or systemd environment:
     ```bash
     systemctl --user set-environment SSS_PASSPHRASE="your-passphrase"
     ```

5. **Keyring Prerequisites for Daemon Mode**
   - Linux (GNOME Keyring / KWallet): keyring session must be available to the Emacs process
   - macOS (Keychain): typically available to GUI apps
   - Alternative: `SSS_PASSPHRASE` environment variable

6. **How It Works**
   - Auto-detection: files containing `⊠{` are automatically opened in sss-mode via magic-mode-alist
   - On open: sss-mode calls `sss open FILE` to decrypt, replaces buffer content with plaintext+markers
   - On save: write-contents-functions handler writes plaintext, then calls `sss seal -x FILE` to re-seal
   - Auto-save and backup files are disabled for decrypted buffers
   - Modeline shows `SSS[sealed]` or `SSS[open]` to indicate buffer state

7. **Key Bindings**
   - Table with all bindings from the mode map:
     | Key | Command | Description |
     |-----|---------|-------------|
     | C-c C-o | sss-open-buffer | Decrypt sealed buffer in place |
     | C-c C-s | sss-seal-buffer | Re-seal buffer and save to disk |
     | C-c C-r | sss-render-buffer | View pure plaintext (markers stripped) in separate buffer |
     | C-c C-i | sss-init | Run `sss init` in project directory |
     | C-c C-p | sss-process | Run `sss seal --project` |
     | C-c C-k | sss-keygen | Run `sss keygen` (generates new keypair) |
     | C-c C-l | sss-keys-list | Show project/keystore keys |

8. **Customisation**
   - `M-x customize-group RET sss RET`
   - `sss-executable`: path to sss binary (default: "sss", searches PATH)

9. **Font-Lock Highlighting**
   - ⊕{} markers: highlighted with sss-open-face (distinct colour for plaintext markers)
   - ⊠{} markers: highlighted with sss-sealed-face (distinct colour for sealed markers)
   - Helps visually distinguish open and sealed content within the same file

10. **Troubleshooting**
    - "sss: command not found" → set sss-executable to absolute path
    - "Decryption failed" → check keystore auth: run `sss open FILE` in terminal first
    - Blank buffer after open → check stderr in *Messages* buffer; likely keystore or passphrase issue
    - "File modified" prompts on kill → expected if buffer was opened but not saved
    - Daemon mode hangs → sss is prompting for passphrase on non-existent TTY; set SSS_PASSPHRASE

11. **Security Considerations**
    - Auto-save disabled: prevents plaintext leaking to #file# auto-save files
    - Backup disabled: prevents plaintext leaking to file~ backup files
    - Brief plaintext window: plaintext exists on disk momentarily during save-then-seal (same as epa-file.el)
    - Buffer content is plaintext in Emacs memory while the buffer is open
